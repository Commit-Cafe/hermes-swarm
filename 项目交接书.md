## 一、项目背景与目标

### 1.1 项目定位

**Hermes Swarm** 是一个多 Hermes-Agent 并行任务编排与可视化监控平台。用户（"监工"）通过 Web 界面同时启动和管理多个 Hermes Agent 实例，实时监控运行状态，自动收集执行结果。

定位：**个人效率工具 + 可展示项目**。

### 1.2 核心需求（用户确认的4个优先功能）

1. **并行任务分发** — 同时启动多个 hermes 进程执行不同任务
2. **实时状态监控** — Web 仪表盘实时查看每个 Agent 的运行状态
3. **结果自动收集** — 执行输出、日志、错误信息自动持久化
4. **成本追踪** — token 消耗和费用统计（目前仅框架就绪，数据采集未完成）

### 1.3 用户的使用场景

用户在公司 WSL2 环境中运行 Hermes Agent（一个 AI Agent 框架），通过 `hermes chat -q "prompt"` 命令行方式执行测试任务。用户希望同时跑多个 hermes 实例（不同模型、不同 skill），并通过 Web 界面统一监控和管理。

## 二、技术架构与选型决策

### 2.1 架构概览

```
[浏览器] ←→ [Next.js 15 前端 :3000]
                ↓ REST/SSE
[FastAPI 后端 :8001]
  ↓ asyncio.subprocess
[WSL2 Ubuntu] → hermes CLI × N（并发受信号量控制）
```

### 2.2 关键技术选型及原因

| 选型 | 原因 |
|------|------|
| FastAPI（非 Flask） | 原生支持 async/await + SSE，适合管理多个 subprocess |
| SQLite（非 PostgreSQL/DynamoDB） | 轻量无依赖，文件级持久化够用 |
| SSE（非 WebSocket） | 单向推送足够，比 WebSocket 更简单可靠 |
| Next.js 15（复用 OAIHUB） | 直接复用现成的仪表盘组件，省去大量前端开发 |
| asyncio.subprocess（非 Docker） | 用户 WSL2 中没有 Docker，hermes CLI 直接可用 |
| base64 编码 prompt | 避免 shell 注入风险（`wsl bash -c` 中的特殊字符问题） |

### 2.3 为什么没用 Pitaya

调研过 Pitaya（tact-lang/pitaya），它的并行编排能力很强，但强依赖 Docker 运行每个 Agent 实例。用户 WSL2 环境没有 Docker，改造 Runner 层工作量大于自写 subprocess 管理。最终决定只借鉴 Pitaya 的策略模式概念（best-of-n、iterative），核心引擎自研。

## 四、核心代码解读

### 4.1 后端核心：hermes_manager.py

这是整个项目最关键的文件，约 270 行。

**HermesProcess 类**（单个任务的生命周期管理）：
- `_build_wsl_command()` — 构建 WSL 命令，通过 base64 编码安全传递 prompt
- `start()` — asyncio.create_subprocess_exec 启动进程，读取 stdout/stderr
- `_read_output()` — 并发读取两个流，每行触发 SSE 事件
- `cancel()` — terminate → kill 进程
- `_update_db_status()` — 根据状态类型更新不同字段

**HermesSwarmManager 类**（全局编排器）：
- `create_task()` — 创建任务记录 + asyncio.Task 异步执行
- `_run_with_semaphore()` — 信号量控制并发（默认最多4个）
- `_emit_global()` — 全局事件广播（给 SSE 用）

**关键设计决策**：
- `swarm_manager` 是模块级单例，后端重启后内存中的运行状态会丢失（但 SQLite 历史保留）
- 事件回调链：HermesProcess → HermesSwarmManager → SSE queue → 浏览器

### 4.2 后端路由：routes.py

- 任务 CRUD（创建/列表/详情/取消）
- 日志查询（按任务+按流类型）
- SSE 事件流（`/tasks/stream`）
- Dashboard 数据（KPIs + 时间序列）
- Agent 状态（运行中数量+活跃进程列表）

### 4.3 前端架构

```
layout.tsx（SidebarProvider + AppSidebar + SiteHeader）
  ├── /dashboard → KPI卡 + 图表 + TaskList
  ├── /tasks     → 创建任务表单
  ├── /agents    → Agent 运行状态卡片
  └── /logs      → 任务选择 + 日志流
```

前端通过 `lib/api.ts` 统一调用后端 API，所有页面都是客户端组件（"use client"），数据通过 3-5 秒定时轮询刷新。

## 六、已知问题与注意事项

### 6.1 环境依赖

| 依赖 | 位置 | 说明 |
|------|------|------|
| Hermes Agent | WSL2 `/root/hermes-agent/` | venv 环境，`source venv/bin/activate` 后可用 |
| hermes CLI | WSL2 PATH | `hermes chat -q "prompt"` 非交互模式 |
| Python | Windows 系统Python 3.12 | 运行后端 FastAPI |
| Node.js | Windows | 运行前端 Next.js |
| WSL2 | Ubuntu 24.04 | 运行 hermes 进程的地方 |

### 6.2 关键注意事项

1. **WSL 命令构建**：`hermes_manager.py` 通过 `wsl bash -c "..."` 执行命令，prompt 用 base64 编码避免注入。如果修改命令构建逻辑，务必注意 shell 特殊字符。

2. **数据库路径**：`data/hermes_swarm.db`，相对路径基于 `backend/` 目录。如果从其他目录启动后端会找不到。

3. **进程管理器是内存单例**：`swarm_manager` 在 `hermes_manager.py` 底部创建。后端重启后运行中的 hermes 进程会变成孤儿进程（数据库中状态停留在 running），需要加启动时的状态恢复逻辑。

4. **CORS 配置**：前端如果在非 3000/3001/5173 端口启动，需要更新 `config.py` 中的 `CORS_ORIGINS`。

5. **Windows 编码问题**：Windows 默认 GBK，hermes 输出的中文和 emoji 可能导致解码错误。`_read_output()` 中用了 `errors="replace"` 处理。

6. **前端端口冲突**：如果 3000 端口被占用，Next.js 会自动使用 3001。需要同步更新 `.env.local` 中的 API 地址和后端的 CORS 配置。

### 6.3 Hermes Agent 相关知识

用户当前的 Hermes 配置：
- **配置文件**：`/root/.hermes/config.yaml`
- **模型**：glm-5（智谱AI）
- **Provider**：zai（通过 `GLM_API_KEY` 环境变量认证）
- **Base URL**：`https://openrouter.ai/api/v1`
- **已安装 Skills**：`python-sort-dict-bugfix`（调试）、`python-line-counter`（DevOps）
- **Profile**：仅 default 一个

Hermes 的多模型切换方式：
```bash
hermes profile create claude-coder --clone    # 克隆配置
hermes profile use claude-coder               # 切换
hermes model                                  # 交互式选择模型
hermes auth add openrouter                    # 添加新的 API Key
```
